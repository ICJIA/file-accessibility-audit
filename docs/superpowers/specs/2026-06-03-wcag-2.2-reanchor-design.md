# Spec — Re-anchor the audit to WCAG 2.2 (with a revert path to 2.1)

- **Date:** 2026-06-03
- **Status:** Approved (design); pending spec review
- **Target release:** v1.24.0 (minor)
- **Author:** cschweda + Claude

---

## 1. Summary

Re-anchor the tool's reference standard from **WCAG 2.1 Level AA** to **WCAG 2.2 Level AA**, surface the new 2.2 criteria honestly (we cannot auto-test them on static PDFs, so they are shown as *not assessed — manual review*, never as auto-failures), add the **IITAA 2.1** standard alongside WCAG + ADA Title II everywhere the pair currently appears, and ship a single **env-overridable version flag** (`WCAG_VERSION`) so the app can revert to WCAG 2.1 — the API on restart, the web UI on rebuild + restart (Nuxt bakes `runtimeConfig.public` at build time, like the existing `REMEDIATION.ENABLED`). A normal redeploy does both.

Two landing-page additions: a **reusable announcement banner** ("a place for new improvements") and a **manager-friendly click-through page** explaining how 2.2 differs from 2.1.

The compliance message stays precise: **IITAA 2.1 and ADA Title II mandate WCAG 2.1 AA; auditing to 2.2 is stricter than required and is optional/forward-looking under IITAA today.**

---

## 2. Background / current state

- `audit.config.ts` is the single source of truth, but **has no WCAG version constant** — `"2.1"` is hard-coded into strings, comments, and `WCAG_CATEGORY_MAP`. Precedent for env-overridable flags exists (`REMEDIATION.ENABLED = process.env.REMEDIATION_ENABLED === "true"`).
- `apps/api/src/services/scoring/conformance.ts` is the binary verdict. It hard-codes `WCAG21/Understanding/...` URLs and "WCAG 2.1 Level AA" headlines, and already has a **`notAssessed` list** — the correct home for new-but-unmeasurable criteria.
- `apps/web/app/utils/wcag.ts` drives the per-finding "WCAG 2.1 References" panel and still cites **4.1.1 Parsing**, which WCAG 2.2 **removes**.
- IITAA is **already** cited in the deeper copy (on-page conformance box in `ScoreCard.vue`, mid-homepage `index.vue`, `data-retention.vue`, and `useReportExport.ts`, including `§E205.4` / `§504.2.2`) but is **missing** from the "headline" surfaces: homepage hero, footer, SEO meta/keywords, and the footer reference-links list.
- The conformance box in `ScoreCard.vue` already links to `iitaa-2-1-standards.html` — the app is correctly on IITAA **2.1** (not 2.0).

## 3. Verified facts (with sources)

### WCAG 2.2 vs 2.1
WCAG 2.2 **adds 9 success criteria** and **removes 1** (4.1.1 Parsing, now obsolete). Of the 9 new, **6 are Level A/AA** and 3 are AAA.

| SC | Name | Level | New? | Understanding slug |
|------|------|-------|------|--------------------|
| 2.4.11 | Focus Not Obscured (Minimum) | AA | new | `focus-not-obscured-minimum` |
| 2.4.12 | Focus Not Obscured (Enhanced) | AAA | new | `focus-not-obscured-enhanced` |
| 2.4.13 | Focus Appearance | AAA | new | `focus-appearance` |
| 2.5.7 | Dragging Movements | AA | new | `dragging-movements` |
| 2.5.8 | Target Size (Minimum) | AA | new | `target-size-minimum` |
| 3.2.6 | Consistent Help | A | new | `consistent-help` |
| 3.3.7 | Redundant Entry | A | new | `redundant-entry` |
| 3.3.8 | Accessible Authentication (Minimum) | AA | new | `accessible-authentication-minimum` |
| 3.3.9 | Accessible Authentication (Enhanced) | AAA | new | `accessible-authentication-enhanced` |
| 4.1.1 | Parsing | (A) | **removed in 2.2** | — |

The **6 new A/AA** criteria are: 2.4.11, 2.5.7, 2.5.8 (AA), 3.2.6, 3.3.7 (A), 3.3.8 (AA). Almost all target **interactive web UI** (focus visibility on scroll, pointer dragging alternatives, 24×24 tap targets, consistently located help, no re-entry of data in a process, no cognitive-test auth). For a **static PDF** they are largely **not applicable / manual**; the few that can touch **interactive PDF forms** are Target Size (2.5.8), Redundant Entry (3.3.7), and Accessible Authentication (3.3.8).

### IITAA
- Current standard is **IITAA 2.1**, effective **June 24, 2024**, which **requires WCAG 2.1 Level AA**.
- Non-web documents (PDFs): **§E205.4** (Level A + AA), with a documented exception releasing non-web docs from 2.4.1, 2.4.5, 3.2.3, 3.2.4. **§504.2** covers authoring tools (the PDF/UA framing).
- **IITAA 2.1 does not reference WCAG 2.2 at all.** → WCAG 2.1 AA is *mandatory*; WCAG 2.2 is *not required* (optional/forward-looking).

**Sources:**
- IITAA — https://doit.illinois.gov/initiatives/accessibility/iitaa.html
- IITAA 2.1 Standards — https://doit.illinois.gov/initiatives/accessibility/iitaa/iitaa-2-1-standards.html
- What's New in WCAG 2.2 (W3C/WAI) — https://www.w3.org/WAI/standards-guidelines/wcag/new-in-22/

> **Must verify at implementation time** (credibility-critical, per the `WCAG_CATEGORY_MAP` comment): that `https://www.w3.org/WAI/WCAG22/Understanding/<slug>.html` resolves for both carried-forward slugs (e.g. `non-text-content`) and the new slugs above. Carried-forward criteria keep their 2.1 slugs.

## 4. Goals / non-goals

**Goals**
1. Make WCAG **2.2 AA** the displayed reference standard, with an honest "superset of the 2.1 AA that IITAA/ADA require" framing.
2. Add **IITAA 2.1** alongside WCAG + ADA Title II on every headline surface ("Full consistency").
3. Document the new 2.2 rules in the **technology overview** (technical) and a new **manager page** (plain-language).
4. Surface form-relevant new criteria as **not assessed — manual review** in the per-document verdict **only when the PDF has interactive form fields** ("balanced-strict").
5. Provide a **single env-overridable flag** to revert everything to WCAG 2.1.
6. Add a **reusable announcement banner** on the landing page.

**Non-goals**
- No new *automated* checks for the 2.2 criteria (infeasible on static PDFs; would be dishonest).
- No change to scoring weights, grade thresholds, or the numeric score.
- No change to the existing 2.1-derived automated failure checks (they carry forward unchanged into 2.2).
- No broad "standards hub" page — the manager page is focused on 2.2-vs-2.1 (may grow later; YAGNI now).

## 5. Design

### 5.1 Config — the revert switch (`audit.config.ts`)

Add a `WCAG` block, env-overridable like `REMEDIATION.ENABLED`:

```ts
export const WCAG = {
  /** Operative reference standard. Flip to "2.1" (set WCAG_VERSION=2.1 in env;
   *  API reverts on restart, web reverts on rebuild + restart) to revert all
   *  labels, links, and the 2.2 not-assessed criteria back to WCAG 2.1. */
  VERSION: ((process.env.WCAG_VERSION === "2.1" ? "2.1" : "2.2")) as "2.1" | "2.2",
  LEVEL: "AA" as const,
  /** Understanding-page base, version-keyed. */
  UNDERSTANDING_BASE: {
    "2.1": "https://www.w3.org/WAI/WCAG21/Understanding/",
    "2.2": "https://www.w3.org/WAI/WCAG22/Understanding/",
  },
  QUICKREF: {
    "2.1": "https://www.w3.org/WAI/WCAG21/quickref/",
    "2.2": "https://www.w3.org/WAI/WCAG22/quickref/",
  },
} as const;

/** The 6 new A/AA criteria introduced in WCAG 2.2 (the AAA ones are listed in
 *  the overview copy but not used by the gate). `pdfFormRelevant` marks the
 *  ones that can apply to interactive PDF forms. */
export const WCAG_22_NEW_AA = [
  { sc: "2.4.11", name: "Focus Not Obscured (Minimum)", level: "AA", slug: "focus-not-obscured-minimum", pdfFormRelevant: false },
  { sc: "2.5.7",  name: "Dragging Movements",           level: "AA", slug: "dragging-movements",          pdfFormRelevant: false },
  { sc: "2.5.8",  name: "Target Size (Minimum)",        level: "AA", slug: "target-size-minimum",         pdfFormRelevant: true  },
  { sc: "3.2.6",  name: "Consistent Help",              level: "A",  slug: "consistent-help",              pdfFormRelevant: false },
  { sc: "3.3.7",  name: "Redundant Entry",              level: "A",  slug: "redundant-entry",              pdfFormRelevant: true  },
  { sc: "3.3.8",  name: "Accessible Authentication (Minimum)", level: "AA", slug: "accessible-authentication-minimum", pdfFormRelevant: true },
] as const;
```

Also add the **announcement** mechanism here (single source of truth, "a place for new improvements"):

```ts
export const ANNOUNCEMENTS = [
  {
    id: "wcag-2.2-2026-06",            // bump id to re-show after dismissal
    badge: "New",
    text: "Now checking against WCAG 2.2 AA. Note: IITAA 2.1 still requires WCAG 2.1 AA as the legal minimum.",
    linkText: "See how 2.2 differs from 2.1",
    linkTo: "/wcag-2-2",
  },
] as const;
```

> `WCAG_CATEGORY_MAP` header comment updated from "WCAG 2.1 SUCCESS-CRITERIA MAP" to note the operative version is config-driven and that these carried-forward criteria are identical in 2.1 and 2.2.

### 5.2 Version-aware plumbing

- **`conformance.ts`**: replace the hard-coded `WCAG21` base in `wcagUrl()` with `WCAG.UNDERSTANDING_BASE[WCAG.VERSION]`; replace "WCAG 2.1 Level AA" in headlines with `WCAG ${WCAG.VERSION} Level AA`. The carried-forward slug map is unchanged (slugs identical across versions). Import from `#config`.
- **`wcag.ts`**: change `U()` to read the base from config (`WCAG.UNDERSTANDING_BASE[WCAG.VERSION]`); the per-finding panel label "WCAG 2.1 References" becomes `WCAG {VERSION} References` (label sourced where it's rendered in `index.vue`). **Remove the obsolete `4.1.1 Parsing`** entry from `pdf_ua_compliance` (replace with a comment noting it's dropped in 2.2; when reverted to 2.1 it's acceptable to omit it since that category is Practical-only and currently unused in the live UI). Keep the existing `1.3.1` entry there.

### 5.3 Conformance gate — balanced-strict (`conformance.ts`)

When `WCAG.VERSION === "2.2"` **and** `qpdf.hasAcroForm` is true, append the **PDF-form-relevant** new A/AA criteria (2.5.8, 3.3.7, 3.3.8) to `notAssessed`, each with a `reason` like *"New in WCAG 2.2. Applies to interactive PDF form controls/processes; not automatically assessed — manual review required."* URLs from the version-keyed base + slug.

- Never added as `failures` (no evidence of failure).
- Not added when the PDF has no AcroForm (keeps static-doc verdicts clean).
- Not added when reverted to 2.1.
- The non-form-relevant new criteria (focus, dragging, consistent help) are **not** added to the per-document verdict — they're web-UI-only and documented in the overview/manager page instead.

### 5.4 Announcement banner (`apps/web/app/components/AnnouncementBanner.vue` — new)

- Reads `ANNOUNCEMENTS` from `#config` (or via runtime config / shared util — match how the web app imports config constants today; verify the existing pattern).
- Renders the newest announcement — **entry index 0** (convention: prepend future announcements to the array) — at the top of `index.vue`, above the hero. Renders nothing if the array is empty or the entry is dismissed.
- **Permanent dismissal (required):** once dismissed, the banner is **never shown again** for that `id` — not on the next page load, reload, or future session. It is shown on the first visit and stays visible across reloads only *until* the user dismisses it.
  - Persist via `localStorage` under `a11y-audit:dismissed-announcements`, holding a **JSON array of dismissed ids** (not a single id), so dismissing one announcement is permanent and independent of any later announcement. Dismiss = append the id and persist.
  - An announcement re-appears **only** when its `id` is new (not in the dismissed array) — e.g. a future improvement, or bumping this entry's `id`.
  - **No flash:** do not render the banner server-side as visible. Render client-side only (e.g. `v-if` flips true after an `onMounted` check of `localStorage`), so a returning, already-dismissed user never sees it during hydration.
  - SSR/`localStorage` access is client-only and wrapped (try/catch + `import.meta.client`/`process.client` guard) so it never throws during SSR or in privacy modes where `localStorage` is unavailable (treat unavailable storage as "not dismissed" but still functional).
- Accessible: `role="status"`, dismiss button with `aria-label="Dismiss announcement"`, keyboard-focusable, meets contrast in dark + light. Badge text ("New") not conveyed by color alone.
- Content-driven, so future improvements are one config entry.

### 5.5 Manager page (`apps/web/app/pages/wcag-2-2.vue` — new, route `/wcag-2-2`)

Plain-language, manager audience:
- What WCAG 2.2 is and how it relates to 2.1 (superset; 2.1 AA carries forward; one criterion removed).
- The **6 new A/AA rules in human terms** + what each means for our PDFs (table: rule, plain description, "applies to our PDFs?" → mostly "interactive UI / rarely for documents", a few "only interactive PDF forms").
- **Why we adopted 2.2 early** — "strict, not loose; we expect IITAA to follow."
- **Compliance bottom line** (call-out box): IITAA 2.1 + ADA Title II require **WCAG 2.1 AA (mandatory)**; **WCAG 2.2 is optional/forward-looking** and not yet referenced by IITAA. Links to IITAA 2.1, W3C "What's New in 2.2", ADA Title II.
- Page meta (title/description), JSON-LD if other pages have it (match pattern), and listed in nav/footer if appropriate.
- Cross-links to/from technical-details.

### 5.6 Technology overview (`apps/web/app/pages/technical-details.vue`)

> Where a label can be config-driven (rendered from `WCAG.VERSION`), prefer that so revert is automatic. Pure prose in static marketing copy can interpolate the version constant or, where that's awkward, be written for 2.2 with a code comment pointing to the flag.

### 5.8 Honesty guardrails (apply to all copy)

1. Never imply the new 2.2 criteria are **automatically tested**. Always "not assessed — manual review" or "manual/interactive."
2. Always keep the legal precision: **IITAA 2.1 + ADA Title II = WCAG 2.1 AA mandatory; 2.2 optional/forward-looking.**
3. "WCAG 2.2 AA is a **superset** of WCAG 2.1 AA" — adopting 2.2 cannot *lower* any existing result.
4. Reverting to 2.1 must leave **no dangling "2.2" strings** in user-facing output (achieved via config-driven labels + a grep check).

## 6. Revert behavior (set `WCAG_VERSION=2.1`)

> **API** reverts on **restart** (tsx re-reads `audit.config.ts`). **Web** reverts on **rebuild + restart** — Nuxt bakes `runtimeConfig.public` at `nuxt build` time (same as `REMEDIATION.ENABLED`), so a bare env change won't flip the front end. A normal redeploy (which rebuilds web) does both. The release notes / README / ops runbook must state this accurately.

- All labels read "WCAG 2.1 Level AA"; all Understanding/quickref links point to `WCAG21/`.
- The 2.2 not-assessed criteria are **not** added to verdicts.
- Per-finding panel reads "WCAG 2.1 References".
- The `/wcag-2-2` page **remains** (informational; harmless under 2.1) and stays linked from the footer/references.
- The announcement banner is **gated on `WCAG.VERSION === "2.2"`**, so a 2.1 revert removes it automatically — no stray "2.2" announcement.

## 9. Risks / mitigations

- **Misreading "2.2 support" as automated coverage** → mitigated by honesty guardrails (§5.8) and "not assessed" framing.
- **Broken/changed W3C slugs** → verify at implementation (§3, §7); fall back to version quickref if a slug is uncertain.
- **Stray "2.2" after revert** → config-driven labels + revert grep smoke test; banner gated on version.
- **Scope creep on the manager page** → keep it 2.2-vs-2.1 focused; no standards-hub.
- **Config import in web** → confirm the existing `#config` / runtime-config pattern the web app already uses before adding `WCAG`/`ANNOUNCEMENTS` reads (don't invent a new import path).
